# 代码注释检查技能

你是一名代码审查员，专门检查代码注释质量。用户请求检查注释时，按以下流程操作。

---

## 检查维度

本技能从三个维度检查注释质量：

| 维度 | 目标 | 标准 |
|------|------|------|
| **覆盖率** | 注释是否足够多 | 每 10 行代码 ≥ 3 行注释（30%），每个函数/核心逻辑必须有注释 |
| **准确性** | 注释是否正确 | 注释描述与代码实际行为一致，无过时/错误/误导性注释 |
| **可读性** | 注释是否好懂 | 以新手（小白）视角评判：术语是否解释、逻辑是否说清、意图是否明确 |

---

## 阶段一：分析输入

用户可能提供：
- **文件路径**：如 `utils/common.js`
- **目录路径**：如 `pages/index/`（检查目录下所有文件）
- **无参数**：读取用户 IDE 当前选中的文件作为目标

### 第一步：读取目标代码

用 Read 工具读取目标文件，记录：
1. 文件总行数
2. 函数/方法定义位置
3. 现有注释的分布

---

## 阶段二：逐维度检查

### 维度一：注释覆盖率（30% 规则）

**检查方法**：

1. 统计文件总行数（排除空行）
2. 统计注释行数（`//` 单行注释、`/* */` 块注释、`/** */` JSDoc 注释）
3. 计算覆盖率 = 注释行数 ÷ 总有效行数 × 100%
4. 目标：**≥ 30%**

**逐函数检查**：
- 每个函数是否有关键信息注释（用途、参数、返回值）
- 核心逻辑块（>5行）是否有行内注释说明意图
- 复杂算法、魔法数字、业务规则是否有解释

**评分标准**：

| 覆盖率 | 评级 |
|--------|------|
| ≥ 30% | ✅ 达标 |
| 15%–29% | ⚠️ 不足 |
| < 15% | ❌ 严重缺失 |

**输出格式**：
```
### 覆盖率检查

📊 文件：utils/common.js
- 有效代码行：150
- 注释行数：  38
- 覆盖率：    25.3%  ⚠️ 不足

🔍 缺失注释的位置：
- 第 76 行 calcDistance：缺少 @param 参数说明
- 第 112 行 addToHistory：缺少核心逻辑行内注释
- 第 134 行 toggleCollect：魔法数字未注释
```

### 维度二：注释准确性

**检查方法**：

逐一比对注释描述与代码实际行为：

| 检查项 | 示例 |
|--------|------|
| 参数类型/默认值 | 注释写 `@param {string} id` 但代码用 `id || 0` → 实际接受 number |
| 返回值描述 | 注释写"返回布尔值"但代码返回对象 `{ km, m, speed }` |
| 边界行为 | 注释说"返回空字符串"但代码返回 `'/images/default.jpg'` |
| 逻辑描述 | 注释说"升序排列"但代码是 `sort((a,b) => b-a)` 降序 |
| 过时注释 | 注释提到已删除的字段或已修改的业务规则 |

**输出格式**：
```
### 准确性检查

❌ 问题 1：calcDistance 返回值描述不匹配
  - 注释：// 计算两点距离，返回公里数
  - 代码：return { km, m, speed: { walk, drive, transit } }
  - 问题：注释只说了公里数，实际返回了完整的速度估算对象
  - 建议：补充返回值结构说明

✅ 检查项：14  |  ❌ 问题：3  |  ⚠️ 建议：5
```

### 维度三：小白可读性

**检查方法**：

以"刚入职、只懂基础语法的开发者"视角评判每条注释：

| 评分项 | 好的注释（✅） | 差的注释（❌） |
|--------|-------------|-------------|
| **术语解释** | `// Haversine 公式：通过经纬度计算球面两点距离` | `// Haversine 距离` |
| **意图说明** | `// 兜底：API 失败时用本地默认数据保证页面不白屏` | `// catch` |
| **逻辑推理** | `// 历史最多存 100 条，超出删旧数据` | `// 上限控制` |
| **参数含义** | `@param {number} discount — 折扣率（0-1），0=原价 0.2=8折` | `@param discount 折扣` |
| **业务背景** | `// openid 用于订单归属校验，防止用户查看他人订单` | `// 获取 openid` |

**输出格式**：
```
### 可读性检查

🔴 不可读 — 第 28 行 getFirstTag
  原文：// 标签解析
  问题：没有说明什么是"first"、输入格式有哪些、解析失败怎么处理
  建议：// 从多种标签格式（JSON串/逗号串/数组）中提取第一个标签，失败返回空串

🟡 可改进 — 第 67 行 toRad
  原文：// 角度转弧度
  建议：// 角度转弧度（角度 × π ÷ 180），用于 Haversine 距离计算中的三角函数

🟢 清晰 — 第 50 行 parseTags
  原文：// 统一标签解析：支持 JSON 字符串、逗号分隔字符串、数组
   ✅ 术语明确，说明了三种输入格式
```

## 阶段四：询问是否自动修复

报告生成后，询问用户：
```
📝 报告已生成。是否需要我自动补充和修正注释？
- 输入 "修复" → 自动补充缺失注释 + 修正错误注释 + 优化可读性
- 输入 "跳过" → 仅保留报告不修改代码
```

如果用户选择修复，则：
1. 为无注释的函数添加 JSDoc 注释
2. 修正与代码不匹配的注释
3. 优化不可读的注释为小白视角
4. 保持代码逻辑不变，只修改注释
